## Цель проекта

Показать современный подход к построению распределённого приложения, которое выдерживает высокую нагрузку и позволяет нескольким командам работать параллельно. Репозитории демонстрируют, как организовать независимые сервисы, обмен данными через очереди и прозрачные процессы деплоя.

## Используемые технологии

- Go — основная платформа для сервисов.
- Apache Kafka — шина обмена сообщениями между всеми компонентами.
- gRPC и HTTP — транспорт между внутренними сервисами и внешними клиентами.
- S3-совместимое хранилище (MinIO) — для хранения медиаданных.
- Docker — упаковка и запуск каждого сервиса.

## Предназначение проекта

Система получает сообщения из Telegram и ВКонтакте, выстраивает очередь обработки, распознаёт текст на изображениях и возвращает подготовленный ответ пользователю. Каждая стадия — отдельный сервис: от forwarder’ов и нормализаторов до message-responder, OCR и финальных отправителей.

## Технические подробности работы

1. Forwarder принимает webhook от мессенджера и публикует событие в Kafka.
2. Normalizer скачивает медиа, складывает его в S3 и отправляет нормализованный JSON дальше в Kafka.
3. Message-responder решает, нужно ли распознавание текста. Если да — формирует задачу для message-responder-ocr, который обращается к doc2text (gRPC) и получает готовый текст.
4. Response-preparer переводит ответ в формат, удобный для конкретного канала, а sender берёт сообщение из Kafka и отправляет пользователю через Telegram/VK API.
5. Все сервисы логируют свои действия, используют единые правила конфигурации и могут масштабироваться независимо друг от друга.
